## 前言

### 这个文档是什么

本文以「短信告警」Android App 的真实开发过程为蓝本，还原了从需求设计到发布准备的全部 12 次提交，展示每个阶段如何与 Claude Code 协作。读者将看到：

- 每个阶段实际使用的 prompt（提示词）
- 每次迭代背后的思考逻辑
- 遇到问题时的排查和修正过程
- 通用的最佳实践和方法论

### 前置知识

本文假设你：

- 了解 Claude Code 是什么，已安装并完成基本配置
- 有 Android 开发基础（Kotlin、Jetpack Compose 等）
- 还没有用 Claude Code 从头到尾完成过一个完整项目

### 项目简介

**短信告警（SMS Alarm）** 是一款 Android 应用，当收到的短信匹配用户预设的关键词规则时，立即以最大音量播放告警声音、同步震动和闪光灯闪烁。适用于服务器告警、紧急通知等场景。

**技术栈：** Kotlin, Jetpack Compose (Material 3), Hilt, Room, DataStore, Coroutines/Flow

**开发周期：** 2026-04-03 ~ 2026-04-05（约 2.5 天，12 次提交）

## 阶段一：需求设计与项目初始化

> 对应提交：`c8d8141` — initial commit（48 个文件，2069 行代码）

### 做了什么

这是整个项目最关键的一次提交。在这一次对话中，Claude Code 完成了：

1. 根据需求描述，生成了详细的设计文档（`docs/superpowers/specs/2026-04-03-sms-alarm-design.md`）
2. 搭建了完整的 Android 项目结构
3. 实现了所有核心功能代码

### 怎么做的

**第一步：写一份详细的需求文档。**

在开始编码之前，我先手动写了一份设计规格文档，包含：

```
请根据以下需求开发一款 Android 短信告警 App：

## 需求描述
当收到包含特定关键词的短信时，立即以最大音量播放告警声音（绕过静音/勿扰模式），
同步进行震动和闪光灯闪烁。告警持续直到用户手动关闭。

## 技术要求
- Kotlin + Jetpack Compose + Material 3
- MVVM 架构，Hilt 依赖注入
- Room 数据库存储规则和短信记录
- DataStore 存储应用设置
- minSdk 26, targetSdk 35

## 功能要求
1. 关键词规则管理：支持多关键词 AND/OR 匹配模式
2. 多种告警方式：音频、震动、闪光灯（可独立开关）
3. 锁屏告警：弹出全屏告警界面
4. 首次引导：引导用户授予所需权限
5. 告警设置：音量、震动、闪光灯开关

[... 详细的数据模型设计、权限列表、UI 设计 ...]
```

**提示：** 需求文档越详细，Claude Code 生成的代码质量越高。关键是把以下几个方面说清楚：

| 维度 | 需要明确的内容 |
|------|----------------|
| 功能 | 具体要做什么、不要做什么、边界条件 |
| 技术栈 | 语言、框架、架构模式、最低版本 |
| 数据模型 | 表结构、字段类型、默认值 |
| UI 设计 | 页面布局、交互方式、显示内容 |
| 权限 | 需要哪些权限、如何引导用户授予 |

**第二步：让 Claude Code 一次性生成项目骨架。**

把设计文档喂给 Claude Code 后，它一次生成了 48 个文件，覆盖：

- Gradle 构建配置
- AndroidManifest.xml（含所有权限声明）
- Room 数据库（Entity、DAO、Converters、AppDatabase）
- Repository 层（含关键词匹配逻辑）
- BroadcastReceiver（SmsReceiver）
- Foreground Service（AlarmService）
- 全屏告警 Activity（AlarmActivity）
- Compose UI（主界面两个 Tab + 权限引导页）
- ViewModel
- 工具类（AudioHelper、VibrationHelper、FlashlightHelper）
- Hilt 依赖注入模块

**第三步：验证生成结果。**

拿到代码后不要直接运行，先检查：

```bash
# 构建验证
./gradlew assembleDebug
```

发现编译问题后，直接告诉 Claude Code 错误信息，让它修复。

### 关键经验

> **「需求文档是你的投资回报率最高的工作。」**
>
> 花 30 分钟写一份清晰的设计文档，比花 3 小时反复修改模糊的 prompt 更高效。Claude Code 可以一次生成数千行高质量代码，前提是你给它足够的上下文。

## 阶段二：核心功能迭代

> 对应提交：`0be68ea` — 添加停止告警的弹框

### 做了什么

初始版本中，停止告警只能通过 `AlarmActivity` 的全屏界面。这次迭代增加了在主界面显示停止告警弹框的能力，并优化了权限引导页。

### 怎么做的

**Prompt 示例：**

```
当前告警触发后，如果用户在主界面（而非锁屏 AlarmActivity），
需要一个方便的停止按钮。请在主界面添加一个停止告警的弹框，
当 AlarmService 正在运行时，主界面顶部显示一个醒目的横幅，
点击后弹出确认对话框，确认后停止告警。
```

### 关键经验

> **「每次只聚焦一个小功能。」**
>
> 每次提交围绕一个明确的改进点。这样做的好处是：
> - prompt 简洁清晰，Claude Code 不容易理解偏差
> - 出问题时容易定位（git diff 清晰）
> - 不容易引入意外改动

## 阶段三：保活机制的探索与试错

> 这是整个项目中最有教育意义的阶段，包含了完整的「尝试 → 发现问题 → 调整」循环。

### 3.1 后台服务常驻

> 对应提交：`27e47d7` — 实现后台服务常驻，手动杀进程后自动恢复监控

**Prompt 示例：**

```
当前的问题是：用户手动杀掉进程后，短信监控就停止了。
需要实现后台服务常驻，确保手动杀进程后能自动恢复监控。
方案：
1. MonitorService 运行在独立进程(:monitor)
2. onTaskRemoved/onDestroy 自愈机制，通过 AlarmManager 自动重启
3. 5 分钟看门狗定时检查服务存活状态
4. BootReceiver 监听开机、屏幕解锁等系统广播
```

**这次改动涉及 11 个文件，新增 357 行代码：**
- 新增 `MonitorService`（独立进程运行）
- 新增 `BootReceiver`（监听系统广播）
- 新增 `RestartReceiver`（接收重启广播）
- 新增 `AlarmState`（全局告警状态管理）
- `SmsReceiver` 改用 Hilt EntryPoint 手动注入

### 3.2 增强保活机制

> 对应提交：`3925e75` — 增强保活机制：WorkManager 定时任务、1 像素 Activity、保活设置引导页

**Prompt 示例：**

```
保活还不够稳定，需要进一步增强：
1. 新增 KeepAliveWorker，通过 WorkManager 每 15 分钟检查并拉起 MonitorService
2. 新增 OnePixelActivity，屏幕关闭时显示 1 像素界面，提升进程优先级
3. 新增保活设置引导页，引导用户开启自启动、电池优化白名单等权限
4. 引导页要能自动检测手机品牌（小米/华为/OPPO/Vivo 等），显示对应的设置路径
```

这次新增了 `KeepAliveWorker`、`OnePixelActivity`、`KeepAliveGuideTab` 等，共 450 行新代码。

### 3.3 发现问题：OnePixelActivity 反而有害

> 对应提交：`51d79f2` — 移除 OnePixelActivity，避免滑动杀进程时被连带杀死

**这是整个项目最重要的一个转折。**

实际测试中发现：`OnePixelActivity` 在部分国产 ROM 上会被系统识别为可见 Activity，导致用户在最近任务中滑动杀掉 App 时，会连带杀死 `:monitor` 进程。

**Prompt 示例：**

```
发现问题：OnePixelActivity 在国产 ROM 上适得其反。
用户在最近任务中滑动杀掉 App 时，OnePixelActivity 导致 :monitor 进程也被连带杀死。
请移除 OnePixelActivity，只保留以下保活机制：
- 前台服务通知
- WorkManager 15分钟定时拉起
- AlarmManager 5分钟看门狗
- onTaskRemoved/onDestroy 自动重启
- BootReceiver 监听多个系统广播
```

### 关键经验

> **「AI 生成的方案不一定是最优的，要测试、验证、调整。」**
>
> Claude Code 提出了 OnePixelActivity 方案，看起来合理，但在国产 ROM 上有反效果。发现后直接告诉它问题所在，它会迅速调整方案。
>
> **「不要怕回退代码。」**
>
> 移除 133 行代码的提交和新增 450 行代码的提交一样有价值。Git 记录了这个试错过程，这正是真实开发的样子。

## 经验总结：与 Claude Code 高效协作的技巧

### 1. 需求文档是第一生产力

| 做法 | 效果 |
|------|------|
| 一句话需求："帮我写个短信告警 App" | 生成代码可能偏离你的想法，需要大量来回修改 |
| 结构化需求：包含功能清单、技术栈、数据模型、UI 设计 | 一次生成高质量代码，后续迭代少 |

**模板建议：**

```markdown
## 项目描述
[一句话描述项目目的]

## 技术要求
- 语言/框架/架构模式
- SDK 版本范围

## 功能清单
1. 功能 A：[具体描述]
2. 功能 B：[具体描述]

## 数据模型
[表名]
| 字段 | 类型 | 说明 |
|------|------|------|

## UI 设计
[页面描述、交互方式]

## 权限需求
[列出所有需要的权限]
```

### 2. 每次提交聚焦一个改进点

对比两种做法：

```
# 差的做法：一次性提多个不相关需求
"请同时添加：1) 规则编辑功能 2) 监控总开关 3) 更换图标 4) 修复权限问题"

# 好的做法：每次一个明确的改进
"请添加告警规则点击编辑功能"
"请添加短信监控总开关"
```

好处：prompt 清晰、diff 易读、出问题好定位。

### 3. 报告问题时要包含上下文

```
# 差的反馈
"保活不工作"

# 好的反馈
"在小米手机上，用户从最近任务中滑动杀掉 App 后，MonitorService 没有自动重启。
日志显示 onTaskRemoved 被调用了但 AlarmManager 没有成功触发。
可能和国产 ROM 的后台限制有关。"
```

### 4. 让 Claude Code 解释它的方案

当你不确定某个方案时，不要盲目接受：

```
"你提出了 OnePixelActivity 方案。请先解释一下这个方案在国产 ROM（小米/华为/OPPO）
上的表现，是否有潜在问题？"
```

如果当时这样做，可能会更早发现 OnePixelActivity 的问题。

### 5. 善用 Git 进行试错

AI 辅助开发的一大优势是速度快，可以大胆尝试：

- 想尝试一个方案？让 Claude Code 实现，然后测试
- 发现不好？`git revert` 或让 Claude Code 回退
- 每个功能点单独提交，保持历史清晰

### 6. 告诉 Claude Code 你的技术偏好

如果你对某些技术选型有偏好，在需求文档中就明确：

```
"使用 KSP（不是 kapt）进行注解处理"
"SmsReceiver 使用手动 Hilt 注入（因为 BroadcastReceiver 不能用 @AndroidEntryPoint）"
"WakeLock 使用 PARTIAL_WAKE_LOCK + ACQUIRE_CAUSES_WAKEUP，10 分钟超时"
```

这些约束会被 Claude Code 精确遵守，减少后续调整。

### 7. 迭代式开发比瀑布式更高效

本项目的开发节奏：

```
第1次提交：完整基础功能（可运行的 MVP）
  ↓
第2次提交：补充停止告警交互
  ↓
第3-6次提交：保活机制（尝试 → 修复 → 调整）
  ↓
第7-11次提交：UI 优化 + 功能完善
  ↓
第12次提交：文档 + 版本号
```

每次提交都产出一个可运行、可测试的版本。这样做的好处是：

- 随时可以在真机上验证
- 问题能被及时发现
- 不会积累大量未验证的代码

### 8. 利用 CLAUDE.md 保持项目上下文

`CLAUDE.md` 文件会在每次对话开始时自动加载到 Claude Code 的上下文中。维护好这个文件，可以让新的对话也能理解项目的全貌。

本项目最终的 CLAUDE.md 包含了：

- 项目概述和技术栈
- 构建命令
- 完整的架构说明和数据流
- 关键设计决策（为什么 SmsReceiver 用手动注入、为什么用 TypeConverters 等）
- 包结构说明
